# Rules and Constraints

Last verified: 2026-07-21

The official source is the
[OpenAI Build Week rules](https://openai.devpost.com/rules). Reverify the live
rules before submission; the official page takes precedence over this summary.

## Hackathon requirements

- Submit by 2026-07-21 at 5:00 PM Pacific Time
  (2026-07-22 at 4:00 AM in Dubai).
- Build a new project during the submission period using Codex and GPT-5.6.
- Enter exactly one category.
- Provide a working, consistently runnable project that matches the demo and
  description.
- Provide a public YouTube demo under three minutes with audio explaining:
  what was built, how Codex was used, and how GPT-5.6 was used.
- Provide the code repository URL.
- Include setup, sample-data, and testing instructions in the README.
- Explain in the README where Codex accelerated the work, which important
  choices remained human decisions, and how Codex and GPT-5.6 contributed.
- Provide the `/feedback` session ID from the task where most core
  functionality was built.
- Keep all submission materials in English or provide English translations.
- A hosted website is optional. A consistently runnable local project is
  acceptable when the repository includes clean setup instructions and sample
  data.

## Repository access and licensing

- Publish the GitHub repository for judging with its source code and project
  documentation licensed under Apache-2.0.
- Keep original sample artwork and the Comic Creator name and visual identity
  outside the Apache-2.0 grant. Mark the sample image directory explicitly.
- Public repository access satisfies the judging-code requirement, so private
  invitations to `testing@devpost.com` and `build-week-event@openai.com` are
  not required while the repository remains public.
- Keep the working project available to the sponsor, administrator, and judges
  free of charge through the judging period.

## Ownership and third-party material

- The submission must be original work owned by the entrant or declared team.
- Do not use copyrighted characters, trademarks, music, fonts, images, data,
  or other assets without the necessary rights.
- Comply with every open-source dependency license.
- Verify authorization and usage terms for every SDK, API, model, and dataset.
- Before selecting an image or AI provider, verify its output rights,
  commercial-use terms, data handling, and suitability for a child-facing
  product.
- Keep third-party notices and attribution where their licenses require them.

## Evidence rules

- Separate implemented-and-tested facts from ideas, plans, and manual QA.
- Preserve dated commits and the primary Codex task as Build Week evidence.
- Do not claim a feature in the README, Devpost description, or video unless it
  works in the submitted build.
- Record important human product, design, and engineering choices in
  `DECISIONS.md`.
